\section{Mises en bouche}

\subsection{}
\begin{minted}{text}
predicate no_3_consecutive_zeros_sub (a:array int) (l:int) =
  ∀ i. 0 ≤ i < l-2 -> ¬(a[i] = a[i+1] = a[i+2] = 0)
\end{minted}
La borne supérieure de la variable $i$ est $l-2$ car en supposant $0 \leq l \leq a.length$, on veut que $0 \leq i+2 < a.length$.

\begin{minted}{text}
predicate no_3_consecutive_zeros (a:array int) =
  no_3_consecutive_zeros_sub a a.length
\end{minted}
En effet, il n'y a pas 3 zéros consécutifs dans $a$ si et seulement s'il n'a pas 3 zéros consécutifs entre les indices $0$ et la taille de $a$.

\subsection{}
\begin{minted}{text}
for i = 0 to a.length-3 do
  invariant { no_3_consecutive_zeros_sub a (i+2) }
  if v = last1 = last2 = 0 then raise TripleFound
\end{minted}
Les bornes de la boucle sont claires: on s'assure que les indices utiles aux tests sont valides. Pour l'invariant, on voit que la fonction lève une exception lorsqu'elle trouve 3 zéros consécutifs entre $i$ et $i+2$. À chaque fin de tour, on a donc qu'il n'y a pas 3 zéros consécutifs entre l'indice $0$ et $i+2$. À la fin, vu les bornes de la boucle, on a bien que $no\_3\_consecutive\_zeros\_sub\ a\ a.length$.

\subsection{}
\begin{minted}{text}
for i = 2 to a.length-1 do
  invariant { last1 = a[i-1] ∧ last2 = a[i-2] }
  invariant { no_3_consecutive_zeros_sub a i }
  let v = a[i] in
  if v = last1 = last2 = 0 then raise TripleFound;
  last2 <- last1; last1 <- v;
\end{minted}
Les bornes de la boucle sont claires: on s'assure que les indices utiles aux tests sont valides.
Pour l'invariant, on voit que la fonction lève une exception lorsqu'elle trouve 3 zéros consécutifs entre $i-2$ et $i$. À chaque fin de tour, on a donc qu'il n'y a pas 3 zéros consécutifs entre l'indice $0$ et $i$. Pour prouver cet invariant, on a besoin d'un second invariant spécifiant que les variables \og last\fg{} contiennent bien ce qu'elles sont sensées contenir (les éléments précédents). À la fin, on a bien que $no\_3\_consecutive\_zeros\_sub\ a\ a.length$.

\subsection{}
\label{q4}
\begin{minted}{text}
for i = 0 to a.length-1 do
  invariant { 0 ≤ count_zeros < 3 ∧ count_zeros ≤ i}
  invariant { i > count_zeros -> a[i-count_zeros-1] ≠ 0 }
  invariant { forall j. 0 < j ≤ count_zeros -> a[i-j] = 0 }
  invariant { no_3_consecutive_zeros_sub a i }
    if a[i] = 0 then
      if count_zeros = 2 then raise TripleFound
      else count_zeros <- count_zeros + 1
    else count_zeros <- 0
done;
\end{minted}
Les invariants deviennent un peu plus compliqués:
\begin{itemize}
\item Le premier permet de borner $count\_zeros$
\item Le second assure que $count\_zeros$ est une sur-approximation de zéros contigus: au delà de la zone délimitée par $count\_zeros$, on a une valeur non-nulle.
\item Le troisième assure que $count\_zeros$ est une sous-approximation de zéros contigus: dans la zone délimitée par $count\_zeros$, on n'a que des valeurs nulles.
\item Enfin, on a l'invariant principal: si aucune exception n'a été levée au rang $i$ c'est que l'on n'a pas rencontré 3 zéros consécutifs.
\end{itemize}

\subsection{}
\label{q5}
\begin{minted}{text}
let rec ghost function num_occ (e:int) (f:int -> int) (i j :int) : int
  variant { j-i }
= if i >= j then 0 else
  if f (j-1) = e then 1 + num_occ e f i (j-1) else num_occ e f i (j-1)
\end{minted}
On \og parcourt\fg{} la fonction de $j$ à $i$, le variant est donc $i-j$. De plus, sans pré-condition sur $j$, on donne un test large d'arrêt: $ i \geq j$.

\subsection{}
\begin{minted}{text}
let count_number_of (e:int) (a:array int) : int
  ensures { result = num_occ e (fun i -> a[i]) 0 a.length}  = ...
\end{minted}
La post-condition permet de s'assurer que la fonction compte bien le nombre d'éléments égaux à $e$ dans tout $a$.

\subsection{}
\begin{minted}{text}
let ref n = 0 in
for i = 0 to a.length-1 do
  invariant { n = num_occ e (fun i -> a[i]) 0 i }
  if a[i] = e then n <- 1+n;
done; n
\end{minted}
L'invariant est ici clair: la référence $n$ compte bien le nombre d'éléments égaux à $e$ entre $0$ et $i$. À la fin de la boucle, on obtient la post-condition.

\subsection{}
\begin{minted}{text}
predicate identical_sub_arrays (a:array int) (o1 o2 l:int)
= ∀ k. 0 ≤ k < l -> a[o1+k] = a[o2+k]
\end{minted}
Deux sous-tableaux sont égaux s'ils le sont à chaque indices.

\subsection{}
On commence par se donner un prédicat $valid\_sub\_array\ a\ o\ l$ spécifiant que le sous-tableau $a[o... (o+l-1)]$ est valide.
\begin{minted}{text}
predicate valid_sub_array (a:array int) (o l:int)
= 0 ≤ o ≤ a.length - l
\end{minted}

On requiert ensuite en pré-condition que les sous-tableaux donnés en argument soient valides.
\begin{minted}{text}
let check_identical_sub_arrays (a:array int) (o1 o2 l:int) : bool
  requires { valid_sub_array a o1 l ∧ valid_sub_array a o2 l }
  ensures  { result = True <-> identical_sub_arrays a o1 o2 l }
= try for k = 0 to l-1 do
    invariant { identical_sub_arrays a o1 o2 k }
    if a[o1+k] <> a[o2+k] then raise DiffFound
  done; ...
\end{minted}
L'invariant suit encore une fois directement la spécification: au rang $i$ de la boucle, on est sûr que les deux sous-tableaux $a[o_1...(o_1+i-1)]$ et $a[o_1...(o_2+i-1)]$ sont égaux.

\section{Takuzu}
\subsection{}
On se donne d'abord un prédicat $no\_3\_consecutive\_identical\_elem'$ exprimant qu'il n'y a pas 3 éléments consécutifs égaux à un paramètre $e$:
\begin{minted}{text}
predicate no_3_consecutive_identical_elem' (acc:int -> elem) (l:int) (e:elem)
= ∀ k. 0 ≤ k < l-2 -> ¬(acc k = acc (k+1) = acc (k+2) = e)
\end{minted}

On instancie ce paramètre ensuite à $Zero$ et à $One$:
\begin{minted}{text}
predicate no_3_consecutive_identical_elem (g:takuzu_grid) (start incr l:int)
= no_3_consecutive_identical_elem' (acc:g start incr) l Zero
∧ no_3_consecutive_identical_elem' (acc:g start incr) l One
\end{minted}

\subsection{}
\begin{minted}{text}
let check_rule_1_for_chunk (g:takuzu_grid) start incr
  requires { g.length = 64 ∧ valid_chunk start incr }
  ensures { rule_1_for_chunk g start incr }
  raises { Invalid -> ¬ (rule_1_for_chunk g start incr) }
= let ref count_zeros = 0 in let ref count_ones = 0 in
  for i = 0 to 7 do
    invariant { 0 ≤ count_zeros < 3 ∧ count_zeros ≤ i}
    invariant { 0 ≤ count_ones < 3  ∧ count_ones ≤ i }
    invariant { i > count_zeros -> acc g start incr (i-count_zeros-1) ≠ Zero }
    invariant { i > count_ones  -> acc g start incr (i-count_ones-1)  ≠ One  }
    invariant { ∀ j. j=1 ∨ j=2 -> count_zeros ≥ j -> acc g start incr (i-j) = Zero }
    invariant { ∀ j. j=1 ∨ j=2 -> count_ones  ≥ j -> acc g start incr (i-j) = One  }
    invariant { no_3_consecutive_identical_elem g start incr i }
    ...
\end{minted}
Beaucoup d'invariants pour cette boucle; ils restent heureusement intuitifs (ils s'inspirent beaucoup de la question \ref{q4}).
\begin{itemize}
\item Les deux premiers donnent des bornes utiles sur les compteurs.
\item Les deux suivants spécifient que les compteurs sont une sur-approximation d'une zone d'éléments contigus: si $i$ est plus grand que le compteur de zéros, on sait que l'élément à l'indice $i - compteur-1$ est différent de zéro.
\item Les deux suivants spécifient que les compteurs sont une sous-approximation d'une zone d'élément contigus: si $j$ est raisonnable et plus petit que le compteur de zéros, on sait que l'élément à l'indice $i - j$ est égal à zéro.\\
  L'hypothèse peu élégante $j=1 \lor j=2$ est nécessaire pour que les prouveurs arrivent à prouver les deux invariants corrects.
\item Enfin, on prouve simplement l'invariant que l'on recherche: au rang $i$, il n'y a pas 3 éléments identiques entre le début du chunk et la position $i$.
\end{itemize}

On note la post-condition exceptionnelle: si une exception est levée c'est que la règle 1 n'est pas valide pour le chunk courant.

\subsection{}
\begin{minted}{text}
let rec function num_occ (e:elem) (g:takuzu_grid) (start incr:int) (l:int)
  requires { g.length = 64 ∧ valid_chunk start incr }
  requires { 0 ≤ l ≤ 8 }
  variant { l }
= if l = 0 then 0 else
  if eq (acc g start incr (l-1)) e
  then 1 + num_occ e g start incr (l-1) else num_occ e g start incr (l-1)

let count_number_of (e:elem) (g:takuzu_grid) start incr : int
  requires { g.length = 64 ∧ valid_chunk start incr }
  ensures { result = num_occ e g start incr 8 }
= let ref n = 0 in
  for i = 0 to 7 do
    invariant { n = num_occ e g start incr i }
    if eq (acc g start incr i) e then n <- n+1
  done; n
\end{minted}

Le code de $num\_occ$ est calqué sur celui de la question \ref{q5}. Ses pré-conditions sont nécessaires à l'accesseur $acc$ et le variant est évident.\\
Les pré-conditions de $count\_number\_of$ sont nécessaires à l'accesseur $acc$ et l'invariant est calqué sur la post-condition.

\subsection{}
\begin{minted}{text}
predicate rule_2_for_chunk (g:takuzu_grid) (start incr:int) =
  num_occ Zero g start incr 8 ≤ 4 ∧ num_occ One g start incr 8 ≤ 4

let check_rule_2_for_chunk (g:takuzu_grid) start incr : unit
  requires { g.length = 64 ∧ valid_chunk start incr }
  ensures { rule_2_for_chunk g start incr }
  raises { Invalid -> ¬ (rule_2_for_chunk g start incr) } =
  if count_number_of Zero g start incr > 4 || count_number_of One  g start incr > 4
  then raise Invalid
\end{minted}

Le prédicat est calqué sur la définition de la deuxième règle.
Les pré-conditions de $check\_rule\_2\_for\_chunk$ sont celles de $count\_number\_of$. La post-condition et la post-condition exceptionnelle sont triviales à la vue de la spécification de $count\_number\_of$.

\subsection{}
\begin{minted}{text}
predicate identical_chunks (g:takuzu_grid) (s1 s2:int) (incr:int) (l:int)
= ∀ k. 0 ≤ k < l -> acc g s1 incr k = acc g s2 incr k ≠ Empty

let check_identical_chunks g start1 start2 incr : bool
  requires { g.length = 64 }
  requires { valid_chunk start1 incr ∧ valid_chunk start2 incr }
  ensures { result = True <-> identical_chunks g start1 start2 incr 8 }
= try for i=0 to 7 do
    invariant { identical_chunks g start1 start2 incr i }
    match acc g start1 incr i, acc g start2 incr i with
    | Zero,Zero | One,One -> ()
    | _ -> raise DiffFound
    end done; ...
\end{minted}
La définition du prédicat $identical\_chunks$ est claire: deux chunks sont identiques si et seulement si ils sont égaux sur chaque indices et différent de $Empty$.
La vérification se fait via une boucle dont l'invariant est justement que les chuncks sont identiques.

\subsection{}
On donne l'explication uniquement pour les colonnes, le prédicat pour les ligne est équivalent.

\begin{minted}{text}
predicate identical_columns (g:takuzu_grid) (s1 s2:int)
= identical_chunks g s1 s2 8 8

predicate no_identical_columns (g:takuzu_grid) (start i:int)
= ∀ k. 0 ≤ k < i ∧ k ≠ start -> ¬(identical_columns g start k)

let check_rule_3_for_column (g:takuzu_grid) (start:int) : unit
  requires { g.length = 64 ∧ 0 ≤ start < 8 }
  ensures { no_identical_columns g start 8 }
  raises { Invalid -> ¬ (no_identical_columns g start 8) }
= for i=0 to 7 do
  invariant { no_identical_columns g start i }
    if i ≠ start && check_identical_chunks g start i 8
    then raise Invalid;
  done
\end{minted}

On se donne un prédicat $no\_identical\_columns\ g\ start\ i$ exprimant le fait qu'il n'y a pas de colonnes identiques dans la grille $g$ entre les indices $0$ et $i$. $check\_rule\_3\_for\_column$ ensuite s'assure que ce prédicat est vrai jusqu'à la huitième colonnes. Si une exception est levée, c'est que le prédicat est faux. Enfin, l'invariant de la boucle est calqué sur ce que l'on veut montrer.

\subsection{}
On ne donne le prédicat que pour la première règle, les autres sont similaires.
\begin{minted}{text}
predicate rule_1_for_cell (g:takuzu_grid) (n:int)
= let cs = column_start_index n in
  let rs = row_start_index n in
  rule_1_for_chunk g rs 1 ∧ rule_1_for_chunk g cs 8
\end{minted}

La règle $i$ est valide pour une cellule $n$ si la règle $i$ est valide pour le chunck représentant la colonne de $n$ et la ligne de $n$.

\subsection{}
\begin{minted}{text}
let check_at_cell (g:takuzu_grid) (n:int) : unit
  requires { g.length = 64 ∧ 0 ≤ n < 64 }
  ensures { valid_for_cell g n }
  raises { Invalid -> ¬(valid_for_cell g n) } = ...
\end{minted}

Avec tout le travail précédent, $check\_at\_cell\ g\ n$ est simple à spécifier: on requiert que la grille $g$ et l'argument $n$ soient raisonnables et on assure en sortie que la grille est valide pour la cellule $n$ si et seulement si l'exception n'a pas été levée.

\subsection{}
\begin{minted}{text}
let check_cell_change (g:takuzu_grid) (n:int) (e:elem) : unit
  requires { g.length = 64 ∧ 0 ≤ n < 64 }
  requires { valid_up_to (g[n<-Empty]) n }
  writes { g }
  ensures { valid_up_to g (n+1) }
  raises { Invalid ->  ¬ (valid_up_to g (n+1)) }
  ensures { array_eq g (old g)[n <- e]}
  raises { Invalid -> array_eq g (old g)[n <- e] }
\end{minted}

La pré-condition $g.length = 64 \land 0 \leq n < 64$ permet de s'assurer que les arguments donnés sont raisonnables.

\begin{itemize}
\item L'assertion est difficile à prouver car on change $g$, il faudrait un lemme de cadre adéquat.
\item La post-condition n'est pas facile à prouver car, à priori, il est difficile de savoir en quoi le fait que $check\_at\_cell$ ne soulève pas d'exception entraîne que le plateau est valide pour $n+1$.
\end{itemize}

\subsection{}
\label{q19}
On donne comme \og lemme de cadre\fg{}:
\begin{minted}{text}
lemma valid_up_to_morphism : ∀ g1 g2 n.
  0 ≤ n < 64 -> g1.length = 64 ->
  array_eq g1 g2 -> valid_up_to g1 n <-> valid_up_to g2 n
\end{minted}

En effet, on sait que $check\_cell\_change$ ne va faire qu'une modification à son entrée, et l'assertion ignore cette modification.\\
Nota Bene:
\begin{enumerate}
\item On utilise le prédicat $array\_eq$ de la bibliothèque standard pour être uniforme avec les spécifications de $solve\_aux$ plus tard: on peut la remplacer ici par l'égalité de Why3.
\item On utilise une équivalence et non une simple implication pour prouver automatiquement des sous-buts de la preuve de $solve\_aux$.
\end{enumerate}

\subsection{}
\label{q20}
On donne comme \og lemme de cadre\fg{}:
\begin{minted}{text}
lemma valid_up_to_extend : ∀ g e n.
  valid_up_to g[n <- Empty] n -> valid_for_cell g[n <- e] n ->
  valid_up_to g[n <- e] (n+1)
\end{minted}

En effet, si une grille est valide jusqu'à l'indice $n$, que la case correspondante est vide et que de plus l'indice $n$ est valide pour une valeur $e$, alors toute la grille où on a mis $e$ à l'indice $n$ est valide jusqu'à l'indice $n+1$.

\subsection{}
\begin{minted}{text}
let rec solve_aux (g:takuzu_grid) (n:int) : unit
  requires { g.length = 64 ∧ 0 ≤ n ≤ 64 }      (* 1 *)
  requires { full_up_to g n ∧ valid_up_to g n }
  writes { g }
  variant { 64-n }                             (* 2 *)
  ensures { array_eq g (old g) }               (* 3 *)
  raises { SolutionFound ->
             extends (old g) g ∧ full_up_to g 64 ∧ valid_up_to g 64 }
\end{minted}
On a modifié le contrat de l'énoncer en trois points:

\begin{enumerate}
\item On requiert que les arguments soient sensés.
\item Le variant est simple: l'appel récursif se fait avec $n+1$ et (sauf quant $n$ a atteint $64$).
\item On a rajouté une post-condition essentielle et non-triviale: si la fonction ne lève pas d'exception, elle \og efface\fg{} les modifications qu'elle a faite sur la grille.
\end{enumerate}
On a en effet besoin de savoir ceci quand les appels récursifs sont infructueux. On utilise ici le prédicat $array\_eq$ de la bibliothèque standard car la structure est bel et bien modifiée dans certains cas (et l'égalité interne de \verb|Why3| n'est donc pas valide).

Pour prouver cette post-condition d'égalité, on a du rajouter deux post-conditions à $check\_cell\_change$ pour indiquer que même si la fonction modifie son entrée, ces modifications ne sont pas visibles.

Enfin, on a rajouté une assertion dans le code de $solve\_aux$:

\begin{minted}{text}
match g[n] with
| Empty ->
  try check_cell_change g n Zero; solve_aux g (n+1)
  with Invalid -> () end;
  assert {valid_up_to g[n <- Empty] n
    by (array_eq g[n <- Empty] (old g))};
  try check_cell_change g n One; solve_aux g (n+1)
  with Invalid -> () end;
  g[n] <- Empty
...
\end{minted}

Celle-ci est nécessaire pour guider les prouveurs dans la marche à suivre.

\subsection{}
Pour prouver la complétude de $solve$, on pourrait ajouter cette spécification:
\begin{minted}{text}
raises { NoSolution -> ¬ (extends (old g) g ∧ valid_up_to g 64) }
\end{minted}
C'est à dire que si $solve$ lève une exception, cela veut dire que la grille n'est pas solvable (et par contraposée, si elle est solvable, l'exception $NoSolution$ ne sera pas levée).

\subsection{}
\verb|Why3| est une bête un peu difficile à appréhender. Pour un utilisateur de \verb|Coq|, le fait de devoir déléguer des preuves non-triviales est assez perturbant, surtout qu'un échec est souvent dû à un manque de lemme intermédiaire peu évident. Mon mode de développement pour les lemmes des dernières questions a souvent été de:
\begin{enumerate}
\item Trouver un lemme non prouvé.
\item Ouvrir \verb|Coq| et comprendre les hypothèses nécessaires à la preuve.
\item Écrire ces hypothèses dans \verb|Why3| et recommencer.
\end{enumerate}
Cela se ressent sans doute dans mes énoncés.

Mise à part ces légers soucis, ce fut un projet intéressant !
